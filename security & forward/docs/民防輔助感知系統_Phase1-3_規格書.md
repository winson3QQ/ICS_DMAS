## 文件說明

本文件定義 Phase 1–3 的開發規格，是 Phase 4（原 v1.4 規格書，完全自主高威脅場景）之前的分階段實作路徑。

每個 Phase 完成後均為**可獨立使用的系統**，不是半成品。Phase 4 是最終目標，本文件不重複描述 Phase 4 內容。

### 核心設計轉變（對照 Phase 4）

| 維度 | Phase 1–3（本文件）| Phase 4（v1.4）|
|------|------------------|----------------|
| Field Node 定位 | **啞節點**（只錄音，AI 集中在 Console） | **智慧節點**（本地 AI，完全自主）|
| 威脅等級 | 演習、訓練、一般任務 | 高威脅場景（網路被切斷、Console 可能失聯）|
| 離線能力 | 資料不遺失，連線後補處理 | 離線時本地 AI 仍能產出草稿 |
| 法律記錄 | AAR / 訓練用途 | 法律效力（YubiKey 簽章、Write-Freeze）|
| STT 消費者 | **指揮部**（事後情資）| 操作員（即時草稿）+ 指揮部（正式記錄）|

## Phase 1：語音 → 逐字稿

**目標**：驗證 STT 可行性，建立最小可用管線。指揮部能看到各組無線電通訊的逐字稿。

### 1.1 設計原則

- **操作員零額外動作**：只按 PTT 說話，放開 PTT，其餘全自動
- **Field Node 只錄音**：不跑任何 AI，不需要向操作員顯示任何資訊
- **資料不遺失**：無論網路狀態，錄音一定寫入本地 SD；連線後自動補傳
- **AI 集中在 Console**：STT 全部在指揮部做，容易維護、容易升級
- **逐字稿給指揮部看**：事後情資與 AAR 用途，不需要即時給操作員確認

### 1.2 系統架構

```
無線電 ──Kenwood audio tap──> Field Node（Pi Zero 2W）
                               ├── PTT 觸發錄音（GPIO 偵測）
                               ├── 寫入 SD 卡（永遠執行，不遺失保障）
                               └── WiFi 上傳（有連線時即時；斷線時 queue）
                                        ↓
                              指揮部 Console（固定）
                               ├── Whisper Large-v3（GPU 加速）
                               └── 逐字稿輸出（時間 + 頻道 + 呼號 + 文字）
```

### 1.3 部署範圍

**Zone 1：固定組**（medical / shelter / security）

- 位置：指揮所 WiFi 範圍內，不移動
- 錄音方式（兩擇一，部署時現場驗證）：
  - **方案 A（優先試）**：指揮所擺接收機（scanner），直接收各組組內頻道 → Console 錄音
  - **方案 B（A 收不到時）**：每組一台 Field Node，tap 組內一台無線電，WiFi 即時回傳
- 涵蓋頻道：各組組內頻道、對指揮部頻道

**Zone 2：前進組**（forward）

- 位置：會移動，可能超出 WiFi 範圍
- 每隊一台 Field Node，tap 隊內一台無線電（錄整個隊頻道）
- 有 WiFi：串流上傳，Console 即時 STT
- WiFi 斷鏈：本地 SD 緩存，重連後自動補傳
- MANET 規劃：Phase 2 實作，Phase 1 先接受斷線期間無即時逐字稿

### 1.4 Field Node 硬體清單

| 元件 | 品牌 / 型號 | 原產地 | 費用（USD）|
|------|------------|--------|-----------|
| 主板 | Raspberry Pi Zero 2W | 英國 | $15 |
| 儲存 | 16GB microSD（A1 級）| 美國（SanDisk）| $5 |
| USB 音效卡 | Plugable USB Audio Adapter | 美國 | $15 |
| 無線電接頭 | Kenwood 2.5mm + 3.5mm 線材 | 美國（Hosa）| $15 |
| 電池 | Mophie Powerstation 10K | 美國 | $40 |
| 殼體 | 簡易防潑水殼（3D 印 / IP54）| — | $10 |
| **合計** | | | **~$100** |

**注意**：Phase 1 不含 LED、4G 模組、GPS、Panic 按鈕。這些屬於 Phase 4。

**儲存容量估算**：16KB/s（8kHz mono 16-bit）× 8hr = 460MB，16GB 可存 ~280 小時，無需擔心容量。

### 1.5 Console 硬體建議

| 項目 | 規格 |
|------|------|
| CPU | Intel N100 或等效（32GB RAM）|
| GPU | RTX 3060 / RTX 4060（8GB VRAM）**必要**：Whisper Large-v3 需要 GPU 才能多路併發 |
| 儲存 | 系統 SSD + 資料 SSD（分離，詳見 §1.8）|
| 音訊輸入 | USB 多頻道音效卡（Zone 1 scanner 方案 A 時）|

**CPU-only 可行性**：N100 跑 Whisper Large-v3 約 1× realtime（30 秒音訊需 30 秒處理）。單路可用，多路 5+ 頻道同時需要 GPU。

### 1.6 錄音策略

```
PTT 按下 → GPIO 偵測低電位 → 開始錄音（寫入 SD）
PTT 放開 → 錄音結束 → 檔案完成

檔案命名：{ISO8601_timestamp}_{channel_id}_{duration_ms}.wav
範例：20260417T143052Z_CH-forward-A_12500.wav

上傳流程：
  有 WiFi → 即時上傳 → Console 收到 200 OK → 移至 /uploaded/
  無 WiFi → 留在 /pending/
  重連時 → 批次補傳 /pending/ 所有檔案
  上傳失敗 → 移至 /failed/，下次連線重試，最多 5 次
```

### 1.7 逐字稿格式

```
[2026-04-17T14:30:52Z] [CH-forward-A] [Alpha 01] 前進至 B 點，人員齊全，請指示
[2026-04-17T14:31:05Z] [CH-forward-A] [Bravo 02] 收到，維持當前位置待命
[2026-04-17T14:31:30Z] [CH-指揮部]   [Alpha 01] 指揮部，Alpha 01 回報，現場已清
```

- **時間**：UTC，來源為 Field Node 錄音開始時間戳
- **頻道**：Field Node 設定的頻道 ID（固定，不從音訊內容推斷）
- **呼號**：從逐字稿內容後處理提取（詳見 §1.9）
- **文字**：Whisper Large-v3 輸出，不自動修改

### 1.8 物理資安（最小化）

| 項目 | 規格 | 理由 |
|------|------|------|
| LUKS2 全磁碟加密 | Console 資料 SSD | 遺失時資料不外洩 |
| 資料彈匣分離 | 系統 SSD / 資料 SSD 物理分開 | 可攜帶、可快速撤離 |
| 其餘（USB 點火、Panic Wipe、YubiKey）| **不做** | Phase 4 實作 |

### 1.9 呼號識別（後處理）

操作員按 SOP 在發話開頭報呼號（例：「Alpha 01 呼叫 Bravo 02」）。

- **Whisper initial_prompt**：每次推論注入呼號清單，提升辨識準確率
  - 範例：`"本次通訊呼號包含：Alpha 01, Alpha 02, Bravo 01, Bravo 02, 指揮部"`
- **正則後處理**：抓取「{呼號} 呼叫」、「{呼號} 回報」等 SOP 格式
- 無法識別時：呼號欄位留空，不猜測

### 1.10 時間同步

- Console 作為 NTP 時間主機
- Field Node 開機時強制對時（timeout 5 秒，失敗以本機 RTC 繼續）
- 誤差容忍：< 1 秒（RTC 漂移 < 1 秒/小時）
- **不做** GPS 授時模組（Phase 4 實作）

### 1.11 待驗證項目（⚠ 必須在 Phase 2 開始前完成）

| ID | 待驗項目 | 驗收標準 | 驗證方法 |
|----|----------|----------|----------|
| **V1-01** | Whisper Large-v3 對類比無線電音質（300–3400Hz 窄頻）的繁中辨識率 | CER < 20% | 實際無線電錄音 20 段，手動比對 |
| **V1-02** | Zone 1 方案 A：指揮所 scanner 能否收到各組組內頻道 | 訊號清晰可辨識 | 現場實測 |
| **V1-03** | Console GPU 多路 Whisper 併發（5+ 頻道）吞吐量 | 平均延遲 < 60 秒 / 段 | 模擬 5 路音訊同時送入 |
| **V1-04** | Field Node PTT 錄音完整性（斷線補傳） | 無資料遺失，補傳成功率 100% | 模擬斷線 8 小時後重連 |

**⚠ V1-01 是最關鍵的驗證。** 若 CER > 40%，需先確認是模型問題還是音質問題（可嘗試改善錄音方式：audio tap 直接取訊號而非麥克風收音），再決定是否繼續。

### 1.12 Phase 1 驗收標準

| 測試項目 | 標準 |
|----------|------|
| STT 準確率（戰術用語，有底噪）| CER < 20% |
| 多路併發延遲（5 頻道）| 每段 < 60 秒 |
| 斷線補傳完整性 | 無遺失 |
| 逐字稿格式正確性 | 時間/頻道/呼號欄位正確率 > 95% |
| Field Node 連續運作 | 8 小時無需人工介入 |

## Phase 2：可靠化 + 多組整合

**目標**：讓系統在真實任務環境中穩定運行，forward 組有 MANET 即時連線，逐字稿整合進指揮部 Dashboard。

### 2.1 MANET（forward 組）

**架構**：BATMAN-adv（Linux kernel module，開源，德國 Freifunk 社群維護）

```
forward 隊伍（7–8 人，隊員間距 < 100m）
    Field Node × 7–8，2.4GHz WiFi mesh（BATMAN-adv）
                ↓
    車載 Relay 站（Pi 4 + 5GHz directional antenna）
    部署位置：隊伍與指揮所中點高處
                ↓
    指揮所 AP → Console
```

**網路降級策略**：

| 狀態 | 行為 |
|------|------|
| WiFi 可達 | 即時串流上傳 |
| WiFi 斷鏈（MANET 斷）| 本地 SD 緩存（Phase 1 基礎行為）|
| 重連 | 自動補傳 |

**不做**：4G 模組、LoRa（Phase 4 可選評估）

**車載 Relay 站硬體**：

| 元件 | 規格 | 費用（USD）|
|------|------|-----------|
| 主板 | Raspberry Pi 4 4GB | $55 |
| WiFi 模組 | USB 3.0 5GHz + 定向天線 | $60 |
| 電源 | 車用 12V DC-DC + UPS HAT | $40 |
| 殼體 | 防水 IP65 | $30 |
| **合計** | | **~$185** |

### 2.2 多頻道並行

- Console Whisper 同時處理 5+ 頻道
- 每頻道獨立 queue，互不影響
- GPU 記憶體規劃：RTX 3060 8GB 可同時跑 2–3 個 Large-v3 實例；多路用批次排程

### 2.3 呼號識別強化

- Whisper initial_prompt 動態更新（任務前輸入當次呼號清單）
- 正則後處理 + 呼號對照表（UUID → 呼號映射）
- speaker diarization（聲紋分離）：Phase 2 評估可行性

### 2.4 時間同步強化

- Field Node 每 5 分鐘對時（Phase 1 只在開機時）
- 斷線期間 RTC 自主計時，重連後校正歷史時間戳

### 2.5 Dashboard 整合

- 逐字稿即時顯示於指揮部 UI（command-dashboard）
- 按頻道分欄、時序排列
- 支援搜尋（呼號、關鍵字、時間範圍）
- API 端點：`GET /api/transcripts?channel=&from=&to=`

### 2.6 上傳可靠性

- SHA-256 hash 冪等性（同一檔案不重複插入 DB）
- /pending/ → /uploaded/ → /failed/ 三層資料夾管理（同 Phase 1，強化重試邏輯）
- Console 端保留原始音檔 72 小時（供 STT 錯誤時手動重跑）

### 2.7 待驗證項目

| ID | 待驗項目 | 驗收標準 |
|----|----------|----------|
| **V2-01** | BATMAN-adv mesh 穩定性（7 人 × 100m 內）| 丟包率 < 5%，吞吐 > 1 Mbps |
| **V2-02** | 車載 relay 覆蓋範圍實測 | 1km line-of-sight 下可用 |
| **V2-03** | 5 路 Whisper 並行延遲 | 每段 < 30 秒（GPU）|
| **V2-04** | 呼號識別率（initial_prompt 注入後）| > 80% 正確提取 |

## Phase 3：AI 輔助分析

**目標**：從逐字稿自動產出戰術摘要與 ICS-214 草稿，減少指揮部行政負擔。

### 3.1 摘要引擎

- **模型**：Llama 3 8B（Q4_K_M）於 Console 本地跑
- **輸入**：單次任務或指定時段的逐字稿
- **輸出**：戰術摘要草稿（各組行動、重要事件、異常回報）
- **信心門檻**：< 0.7 自動標記「⚠ 需人工確認」
- **原則**：AI 產出草稿，指揮官審核後方可作為正式紀錄

### 3.2 ICS-214 自動草稿

| ICS-214 欄位 | 來源 |
|-------------|------|
| 日期 / 時間 | 系統自動（log_time）|
| 單位名稱 | 頻道 ID 對照表自動映射 |
| 活動記錄 | Llama 3 8B 摘要草稿（需人工審核）|
| 人員簽署 | **人工**（Phase 3 不做數位簽章，Phase 4 再加）|

### 3.3 矛盾偵測

- **實作方式**：規則引擎（確定性邏輯，不依賴 LLM）
- **範例規則**：
  - 同頻道 5 分鐘內出現「前進至 B 點」和「撤離 B 點」→ 標記矛盾
  - 同一呼號回報位置與其他呼號的位置描述不一致 → 標記
- **原因**：LLM 做矛盾偵測準確率不穩（Phase 2 E2B 實測 0/5），規則引擎更可靠

### 3.4 人工審核流程

```
Llama 草稿生成 → 顯示於 Dashboard（標記「草稿，未審核」）
      ↓
指揮官審閱 + 修改
      ↓
確認送出 → 寫入正式紀錄（無數位簽章，Phase 4 加）
```

### 3.5 待驗證項目

| ID | 待驗項目 | 驗收標準 |
|----|----------|----------|
| **V3-01** | Llama 3 8B 戰術摘要品質（人工評估）| 摘要可讀、關鍵事件不遺漏 |
| **V3-02** | N100 + GPU 同時跑 Whisper + Llama | 無 OOM，延遲可接受 |
| **V3-03** | 矛盾規則引擎準確率 | 偵測率 > 80%，誤報 < 20% |

## 各 Phase 與 Phase 4 的關係

```
Phase 1 ──驗 STT 可行性──> 逐字稿管線（最小可用）
   ↓
Phase 2 ──可靠化──> MANET + 多組 + Dashboard 整合
   ↓
Phase 3 ──AI 分析──> 摘要 + ICS-214 草稿
   ↓
Phase 4 ──完全自主──> 本地 AI + 法律記錄 + 物理資安強化
          （見 民防輔助感知系統_規格書_v1_4.md）
```

Phase 4 的功能在 Phase 1–3 中**刻意不做**，原因：

| Phase 4 功能 | 不在 Phase 1–3 的原因 |
|-------------|----------------------|
| Field Node 本地 Whisper Tiny | Phase 2 E2B 實測 CER 45.3%，目前技術不成熟 |
| YubiKey 數位簽章 / Write-Freeze | Phase 1–3 定位為訓練/演習，無法律記錄需求 |
| USB 點火鑰匙 / Panic Wipe / LUKS2（完整版）| 威脅模型未達 Phase 4 等級 |
| 4G 模組 | MANET 斷了就斷了，接受延遲 |
| GPS 授時 | NTP 足夠 |
| LoRa 低頻寬備援 | 威脅模型未達需要 |
