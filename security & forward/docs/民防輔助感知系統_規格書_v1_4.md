## 專案概述 (Project Overview)

本系統專為民防團體設計，是一套輕量化、離線優先且具備高度資安防護的分散式指管紀錄工具。v1.1 在 v1.0 的指揮部架構基礎上，新增完整的「前線節點 (Field Node)」規格，實現從前線通訊員到指揮部的全自動語音轉文字記錄鏈。

### 1.1　核心設計原則

- **零操作負擔 (Zero Cognitive Load)：** 前線操作員唯一的動作是按 PTT 說話，系統全程自動運作。
- **離線優先 (Offline First)：** 任何節點在斷網環境下均可獨立運作，網路恢復後自動同步。
- **完整追溯 (Full Traceability)：** 每一筆通訊均附時間戳記、裝置 ID、頻道標識，支援事後搜尋與匯出。
- **物理資安 (Physical Security)：** USB 點火鑰匙、一鍵銷毀、資料彈匣分離，確保實體威脅下的資料保護。
- **每人覆蓋 (Per-Operator Coverage)：** 每位隊員（含 RTO）各配一台 Field Node，無通訊盲點。

## 前線節點規格 (Field Node Spec)

本章為 v1.1 核心新增章節，定義每位前線操作員配備的自動錄音裝置完整規格。

> **v1.5 架構變更：** Field Node 定位為**純錄音 + 上傳節點**，不執行任何本地 AI 推論（STT/LLM）。所有語音辨識由 Console Whisper Large-v3 集中處理。原因見 §5.1。

### 3.1　設計原則

- 操作員唯一動作：按 PTT 說話，放開 PTT，其餘全自動。
- 裝置全程靜默運作，不發出聲音，不需要操作員介入。
- 斷網時自動暫存**音檔**，網路恢復後自動上傳至 Console，任務資料不遺失。
- 體積最小化，整合於現有胸掛 / 腰掛系統，線材完全內藏。

### 3.3　殼體與整合設計

Field Node 整合於操作員現有的胸掛或腰掛系統，採用 MOLLE 相容殼體直接固定，與無線電並排。所有線材完全內藏，零外露。

Kenwood 標準接頭（2.5mm + 3.5mm 雙插）各腳位定義：

| 接頭 | 腳位 | 用途 |
|------|------|------|
| **3.5mm** | Tip（尖端） | 音訊輸出（接音效卡錄音輸入） |
| **3.5mm** | Sleeve（外環） | 接地 (GND) |
| **2.5mm** | Ring（中環） | PTT 偵測（按下時短路至 GND）→ 接 GPIO4 |
| **2.5mm** | Tip（尖端） | 麥克風輸入（本版本暫不使用） |

- **精確：** PTT 信號代表操作員主動發話，不會誤判無線電底噪或靜電為語音。
- **節能：** 僅在 PTT 期間啟動錄音，大幅降低 MicroSD 寫入與運算負擔。
- **可靠：** 硬體信號，不受語音內容或音量影響。

### 3.5　LED 指示燈設計

| LED | 顏色狀態 | GPIO 行為 | 說明 |
|-----|----------|-----------|------|
| 🟢 **電源** | 常亮綠 / 閃紅 | GPIO 17：常亮=正常，1Hz 閃爍=電量低於 20% | 讓操作員隨時知道電量狀態 |
| 🔵 **網路** | 常亮藍 / 熄滅 | GPIO 27：常亮=已連線上傳，熄滅=離線暫存中 | 確認資料有無成功上傳指揮部 |
| 🔴 **錄音** | PTT 期間亮紅 | GPIO 22：PTT 信號觸發即亮，放開即滅 | 操作員確認系統正在錄音 |

- LED 亮度：戶外白天可視；夜間模式（實體開關）調暗至 10% 亮度，避免暴露位置。
- **電源 LED 附加容量警告（I-03 修正）：** MicroSD 可用空間 < 10% 時，電源 LED（GPIO 17）以 2Hz 快速閃爍（有別於電量低的 1Hz 閃爍），提示操作員及時清理或更換儲存媒體；同時停止接受新錄音，直至可用空間恢復 > 10%。
- **電量監測模組（G-01 修正）：** Mophie Powerstation 10K 無數位輸出介面，Raspberry Pi Zero 2W 無內建 ADC，因此新增 INA219 電流/電壓感測模組（I2C 介面，選配，約 $3 USD），掛接於 Mophie → Pi Zero 2W 供電線路，實時量測電壓推算剩餘電量，作為電源 LED 1Hz 閃爍的判斷依據。
- 實體開關：殼體側面加裝小型撥動開關，可完全關閉所有 LED（無光靜默模式）。
- 每 30 秒嘗試連線指揮部 Console API（心跳包確認連線狀態）。
- 有網路時：批次上傳 /pending/ 所有檔案，收到 200 OK 後移至 /uploaded/。
- 無網路時：繼續錄音暫存至 /pending/，LED 藍燈熄滅提示離線狀態。
- 上傳失敗時：移至 /failed/，下次連線時重試，最多 5 次，超過則保留等待人工處理。
- 任務結束：如長期無網路，可直接取出 MicroSD 帶回指揮部手動匯入。

## 資安機制 (Security) — 強化版

### 4.1　時間同步

- 指揮部 Console 為時間主機（Time Master），加裝 GPS 授時模組（u-blox NEO-M8N）。
- Field Node 開機時強制對時，任務期間每 5 分鐘對時一次。
- **開機對時 Fallback（G-02 修正）：** Field Node 開機時若 WiFi 未連線，對時請求逾時（timeout: 5 秒，retry: 3 次），系統繼續以本機 RTC 時間運作，網路 LED 保持熄滅；首次連線成功後立即強制對時並更新所有 /pending/ 檔案的時間戳記。對時失敗不阻塞開機流程。
- 斷網環境下，GPS 授時模組獨立提供精確時間，精度 < 1 秒。
- 所有 log_time 欄位記錄 UTC 時間，本地時區轉換僅在顯示層處理。

### 4.2　強化版 Panic Wipe

| 層級 | 機制 | 說明 |
|------|------|------|
| **第一層** | Streamlit 前端按鈕（有保護蓋 UI） | 按鈕需先點擊「解除保護」才會出現確認選項 |
| **第二層** | 文字輸入確認 | 操作員必須手動輸入「DESTROY-{裝置代號}」才能繼續 |
| **第三層** | YubiKey 實體授權 | 必須插入 YubiKey 並通過 FIDO2 驗證，缺一不可 |
| **硬體備援** | 實體硬體觸發鈕（保護蓋遮蔽） | 僅供軟體失效時使用，需破壞保護蓋才能觸發，留下實體記錄 |

- 執行結果：`cryptsetup luksErase` 抹除 OS 碟與 Data 碟 Header，0.5 秒內資料永久遺失。
- Field Node 亦具備簡化版 Panic：長按實體按鈕 5 秒，觸發 MicroSD 快速格式化。
- **Panic 按鈕規格（I-05/I-06 修正）：** Panic 按鈕為獨立實體按鈕，非 Kenwood PTT 複用，建議採用帶防誤觸保護蓋的開關。按鈕接至 GPIO 5（或其他未佔用的 GPIO，需在 I-01 腳位確認後決定）。PTT 按鈕（GPIO 4）與 Panic 按鈕（GPIO 5）分屬不同電路，軟體層以長按 5 秒邏輯偵測 Panic，消除戰場誤觸風險。

### 4.3　三層備份策略

- **第一層（即時）：** SQLite WAL 模式，每次寫入自動 Journal，減少損毀風險。
- **第二層（每小時）：** Console 自動將 Data Cartridge 加密備份至第二顆 USB 3.0 SSD（備援彈匣）。
- **第三層（任務結束）：** 完整備份至離線加密硬碟，異地保管。
- **Field Node 層：** 每台裝置 MicroSD 保留 /uploaded/ 備份，至少保留 72 小時的原始資料。

### 4.4　YubiKey 雙人備援

- 每套系統預備兩把 YubiKey，主用一把，備用一把由另一位授權人員保管（雙人原則）。
- 啟用雙人備援密碼時，需兩位授權人員同時在場，各輸入自己的半段密碼。
- 所有備援操作自動記錄稽核日誌，時間戳記 + 操作者 UUID + 操作內容，不可刪除。

## AI 引擎規格 (AI Engine)

### 5.1　Field Node 本地 STT 不可行分析（v1.5 定案）

**結論：Field Node 不執行任何本地 STT 推論。所有語音辨識集中於 Console。**

Field Node 跑本地 STT 需要同時滿足兩個條件：**準確率夠高**且**延遲夠低**。這是兩個獨立的問題，分別受不同瓶頸限制，而 Field Node 兩個都無法解決。

#### 問題一：準確率不足 — Whisper 訓練資料的根本限制

| 瓶頸 | Whisper 繁體中文訓練語料不足 |
|------|---------------------------|
| 能靠硬體升級解決嗎？ | **不能。** 準確率由模型決定，與硬體無關 |
| 能靠 fine-tune 解決嗎？ | **理論上可以，** 但需要大量繁中語音標註資料 |

ICS_DMAS Phase 2 在 Pi 500（規格高於 Pi Zero 2W）實測所有 Whisper 模型：

| 模型 | CER（字元錯誤率） | 問題 |
|------|-------------------|------|
| Whisper Tiny | 52.8% | 「止血帶」→「只血代」、「建物倒塌」→「見悟到他」 |
| Whisper Base | 48.1% | 略優於 Tiny，仍不可用 |
| Whisper Small | 43.2% | 同上 |
| Whisper Medium | 37.5% | 最佳，仍遠超 15% CER 門檻 |

- **所有模型輸出為簡體中文**，無法切換繁體
- 戰術用語（「回報」「據點」「撤離」）與醫療術語同樣錯誤率高
- **這是 Whisper 訓練資料的根本限制**：繁體中文語料不足，台灣口音/用語覆蓋率低

參考：`command-dashboard/tests/phase2_e2b/BENCHMARK_REPORT.md §8`

#### 問題二：延遲太高 — Memory bandwidth 的物理天花板

| 瓶頸 | Memory bandwidth 決定推論速度（tok/s ≈ bandwidth ÷ 模型大小） |
|------|-----------------------------------------------------------|
| 能靠 fine-tune 解決嗎？ | **不能。** 延遲由硬體決定，與模型訓練無關 |
| 能靠硬體升級解決嗎？ | **能，但不在 Field Node 上。** N100（~76 GB/s）可解決 |

| 規格 | Pi Zero 2W | Pi 500 | N100 Console |
|------|-----------|--------|-------------|
| RAM | 512MB | 8GB | 32GB DDR5 |
| Memory bandwidth | ~6 GB/s | ~34 GB/s | ~76 GB/s |
| Whisper Tiny 推論 | 不可行（RAM 不足） | 可跑但延遲過高 | 預估可即時 |

Phase 2b 在 Pi 500 上測試所有模型（0.8GB ~ 7.2GB）延遲均超標 3-15 倍。Pi Zero 2W 規格僅為 Pi 500 的 1/5 ~ 1/6，完全無法執行任何即時 AI 推論。

**即使 fine-tune Whisper 解決了準確率，Pi Zero 2W 的 6 GB/s bandwidth 仍然是不可突破的物理天花板，延遲問題不會改善。** 反之，即使升級到 N100 解決了延遲，沒有 fine-tune 的 Whisper 繁中準確率仍然不夠用。兩個問題必須同時解決，而 Field Node 兩個都解決不了。

參考：`command-dashboard/tests/phase2_e2b/BENCHMARK_REPORT.md §9.6-9.7`

#### 問題三：AI 加速器無法解決上述任一問題

- **Hailo-8L NPU**：為 CNN/視覺模型設計，不支援 transformer 架構（Whisper 為 transformer），on-chip SRAM ~2MB 不足以存放模型權重
- **Coral Edge TPU**：同為 CNN 加速器，不支援 Whisper/LLM
- **Pi Zero 2W 無 PCIe 介面**：無法安裝任何 AI 加速卡
- 即使未來有支援 transformer 的邊緣加速器，**準確率問題仍需 fine-tune 解決**，加速器只能解延遲

#### 兩個問題的解決路徑

| 問題 | 解決方案 | 在哪裡解決 |
|------|---------|-----------|
| 準確率（CER 45%+） | fine-tune Whisper 或改用雲端 STT API | 模型層（與硬體無關） |
| 延遲（推論太慢） | 升級到 N100（bandwidth ~76 GB/s） | 硬體層（與模型無關） |

**兩者必須同時解決，STT 才可用。Field Node 兩個都解決不了，因此 STT 必須放在 Console 或雲端。**

#### 定案架構

| 部署位置 | 功能 | 說明 |
|----------|------|------|
| **Field Node（Pi Zero 2W）** | 純錄音 + 暫存 + 上傳 | PTT 觸發錄音，存 WAV/OGG 至 MicroSD，有網路時自動上傳 |
| **Console（N100 Mini PC）** | Whisper Large-v3 (int8) STT | 接收音檔後集中推論；解決延遲問題，準確率待 N100 到貨後實測 |
| **雲端 API（備援/主力）** | Google/Azure/Gemini STT | 同時解決準確率 + 延遲；Console 繁中準確率不足時作為主力方案 |

> **離線情境處理：** Field Node 斷網時持續錄音暫存於 MicroSD（/pending/），操作員無法即時看到文字轉錄。網路恢復後自動上傳至 Console 進行 STT。最壞情況：任務結束取出 MicroSD 帶回 Console 手動匯入。

### 5.2　效能基準測試計畫（v1.5 更新）

- 測試環境：Intel N100 Mini PC（Console），DDR5 雙通道 32GB。
- 測試素材：20 段實際無線電錄音，各 10-30 秒，涵蓋中文、台語、英文戰術用語。
- ~~**Field Node 驗收標準：**~~ v1.5 移除，Field Node 不跑 STT。
- **Console 驗收標準：** 10 個 30 秒音檔同時上傳，平均 Turnaround Time < 20 秒。
- **無線電抗噪測試：** 在有底噪（S5 以上雜訊）環境下，Whisper Large-v3 中文辨識率 > 85%。
- 如 N100 無法達標，評估升級至 i5 或改用雲端 STT API。

### 5.3　ICS-214 自動生成邏輯

| ICS-214 欄位 | 來源 | 說明 |
|-------------|------|------|
| **日期 / 時間** | 系統自動填入 | 從 log_time 欄位取得，不依賴 AI |
| **單位名稱** | 裝置 UUID 對照表 | 自動映射，不依賴 AI |
| **活動記錄 (Activity)** | Llama 3 摘要 | AI 產生草稿，人工審核後方可簽署；信心分數 < 0.7 時自動標記需人工確認 |
| **人員簽署** | YubiKey FIDO2 授權 | 簽署後 Write-Freeze，不可修改；簽署人 UUID 記錄於資料庫 |

> ⚠ **重要原則：AI 摘要為草稿，指揮官必須審核後才能簽署。系統不允許未審核的 AI 輸出直接成為正式記錄。**
