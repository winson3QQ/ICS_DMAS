**Pi 500 不適合跑即時 AI 推論。** 其角色應定位為 WebSocket server、PWA host、Pi push、錄音暫存。

### 9.2 語音輸入的可行路線

| 路線 | 方式 | 優點 | 缺點 |
|------|------|------|------|
| A. 雲端 API | Gemini 2.5 Flash（音檔→JSON 一步到位） | 快、準、免 STT | 需要網路 |
| B. N100 Console | Pi 錄音 → 送 N100 跑 LLM | 離線可用、準確率驗證過 | 需硬體、仍有延遲 |
| C. 規則引擎 | 操作員照格式口述 + 關鍵字解析 | 零延遲、離線、零成本 | 需嚴格口述格式 |
| D. 維持手動 | 不做語音輸入 | 最穩定 | 現場速度慢 |

### 9.3 延遲瓶頸分析

延遲的根因不是記憶體容量，而是 **memory bandwidth**：

```
每 token 時間 ≈ 模型載入大小 ÷ memory bandwidth
```

Pi 500 LPDDR4X-4267 bandwidth ≈ 34 GB/s。Gemma4 E2B 載入 7.2GB，每 token 需掃描全部權重：
- 理論：7.2GB ÷ 34 GB/s = 0.21s/token → 4.7 tok/s
- 實測：4.2 tok/s（接近理論值）

**Pi 500+ (16GB LPDDR4X) 無法解決**：bandwidth 相同，decode 速度不變。只能消除 swap 造成的偶發暴增。

**Gemma4 E2B 的 MoE 架構問題**：雖然每次推論只啟動 2.3B 參數，但整個 5.1B（7.2GB）權重都要載入記憶體並逐 token 掃描。密集模型（非 MoE）載入多少就用多少，不浪費 bandwidth。

2. **實測 tok/s 遠低於理論預估**：Gemma3 4B 理論 ~14 tok/s，實測 ~2.8 tok/s。Ollama 在 aarch64 上的推論效率約為理論值的 20-30%
3. **所有模型矛盾偵測 0%** — 確認矛盾偵測應改用規則引擎
4. **AI 加速器（Hailo-8L）無法解決 LLM 問題** — Hailo 為 CNN/視覺設計，不支援 transformer 架構；on-chip SRAM ~2MB 不足以存放 LLM 權重；Pi 5 PCIe Gen3 x1（~1 GB/s）也會成為瓶頸

### 9.7 Phase 2 + 2b 最終結論

**Pi 500 不適合跑任何本地 LLM 即時推論。** 所有測試過的模型（0.8GB ~ 7.2GB）延遲均超過 15 秒門檻 3-15 倍。

**Pi 500 的正確角色**：WebSocket server、PWA host、Pi push 資料管線、錄音暫存上傳。不跑 AI。

**AI 推論架構定案**：

| 功能 | 方案 | 硬體 |
|------|------|------|
| 結構化輸出 | Phi4-mini（2.5GB, Q4）或 Gemini 2.5 Flash API | N100 Console 或雲端 |
| STT | **Breeze ASR 25（CTranslate2 int8, 1.5GB）** 或雲端 API | **N100 Console**（正規化 CER 7.5% PASS）或雲端 |
| 矛盾偵測 | 規則引擎（確定性邏輯） | 任何硬體 |
| 視覺監控 | Pi 5 + Hailo-8L | 邊緣（正確用法） |

**N100 Console 最低建議規格**：DDR5 雙通道 32GB（bandwidth ~76 GB/s），Phi4-mini 2.5GB 預估 ~30 tok/s，到貨後需實測。
