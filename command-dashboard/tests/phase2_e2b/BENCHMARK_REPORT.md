## 7. Ollama 調參最佳化測試（2026-04-17）

建立 `gemma4-fast` Modelfile（num_ctx=1024, num_batch=128, num_predict=200, temperature=0, top_k=1），測試軟體調參能否降低延遲。

| Case | gemma4:e2b（原始） | gemma4-fast（調參後） | 變化 |
|------|-------------------|---------------------|------|
| STRUCT-01 | 125.0s | 42.5s | 2.9x 快（原本撞 swap） |
| STRUCT-02 | 41.0s | 49.2s | 略慢 |
| STRUCT-03 | 30.8s | 40.7s | 略慢 |
| STRUCT-04 | 50.9s | 55.9s | 持平 |

**Decode 速度穩定在 4.2 tok/s** — 這是 Pi 500 memory bandwidth 的硬上限。每次回應 140~200 tokens，光 decode 就要 33~48 秒，軟體調參無法突破 15 秒門檻。

STRUCT-01 大幅改善是因為縮小 context window 避免了 swap，但對其他案例效果有限。

### 8.2 各模型大小比較（STT-01）

| 模型 | 大小 | 載入時間 | 推論延遲 | 辨識品質 |
|------|------|---------|---------|---------|
| Tiny | ~75MB | 0.9s | 6.8s | CER ~63%，幾乎不可用 |
| Base | ~140MB | 7.7s | 12.5s | 稍好但醫療術語仍全錯 |
| Small | ~460MB | 24s | 39s | 略有改善但仍不及格 |
| Medium | ~1.5GB | 90s | 90s | 最好但延遲不可接受 |

### 8.3 典型錯誤

所有模型的共同問題：
- **繁→簡轉換**：輸出一律為簡體中文
- **醫療術語全面崩壞**：「建物倒塌」→「見悟到他」、「GCS十五」→「機械食物」、「止血帶」→「只血代」、「骨折」→「股者」
- **根因**：Whisper 中文訓練資料以簡體為主，且缺乏醫療領域語料

**結論：Whisper 在 Pi 500 上做繁體中文醫療 STT 不可行。不是模型大小問題，是訓練資料的根本限制。**

## 9. 最終結論與建議

### 9.1 Pi 500 AI 能力總判定

| 能力 | 測試結果 | 可用性 |
|------|---------|--------|
| LLM 結構化輸出 | 準確率 93.5%（PASS），延遲 40~50 秒（FAIL） | 非即時場景可用 |
| LLM 矛盾偵測 | 0%（FAIL） | 不可用 |
| LLM 調參最佳化 | decode 4.2 tok/s 為硬體天花板 | 無法突破 |
| STT (Whisper) | CER 45.3%（FAIL），醫療術語全錯 | 不可用 |

**Pi 500 不適合跑即時 AI 推論。** 其角色應定位為 WebSocket server、PWA host、Pi push、錄音暫存。

### 9.2 語音輸入的可行路線

| 路線 | 方式 | 優點 | 缺點 |
|------|------|------|------|
| A. 雲端 API | Gemini 2.5 Flash（音檔→JSON 一步到位） | 快、準、免 STT | 需要網路 |
| B. N100 Console | Pi 錄音 → 送 N100 跑 LLM | 離線可用、準確率驗證過 | 需硬體、仍有延遲 |
| C. 規則引擎 | 操作員照格式口述 + 關鍵字解析 | 零延遲、離線、零成本 | 需嚴格口述格式 |
| D. 維持手動 | 不做語音輸入 | 最穩定 | 現場速度慢 |

### 9.3 建議策略

1. **矛盾偵測**：規則引擎實作（確定性邏輯，不需 LLM）
2. **語音輸入**：雲端 API（路線 A）為主，離線時 fallback 規則引擎或手動輸入
3. **Pi 500 上不再追求本地 AI 推論**，專注其核心職責
