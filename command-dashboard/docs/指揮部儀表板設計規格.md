## v2.0 變更說明（對應 cmd-v0.10.2）

### 事件視覺一致性

1. **Popup 事件按鈕左色條**：長按地圖選事件類型時，每個按鈕左側顯示 severity 色條（紅/黃/灰），選之前即可預判嚴重程度。Hover 背景也跟隨 severity 色系。
2. **Sidebar 事件卡片左色條**：改為 severity 色（取代原本的單位識別色）。已結案卡片同樣採 severity 色但 opacity 0.45 淡化。
3. **Sidebar 群組標頭**：按組別分類時，組別標題改為中性灰（`var(--text3)`），顏色語義完全交給 severity 色條。

### 事件卡片資訊密度

4. **卡片第二行加入 assigned unit**：格式 `回報單位 → 處理單位 EV-XXX`。中間 `flex:1` 在寬版自然展開顯示，窄版（20% sidebar）自動縮到 0 不破壞排版。
5. **Sidebar expanded 寬度**：40% → 30%，地圖保留 50%，比例更平衡。

### 節點 Modal 事件頁

6. **雙來源事件顯示**（shelter / medical）：事件頁分三區塊：
   - 「本組處理中」：`assigned_unit === unit` 且非本組回報的 ICS 事件（優先顯示）
   - 「本組回報」：`reported_by_unit === unit` 的 ICS 事件
   - 「PWA 組內事件」：非同步從 `/api/pi-data/{unit}/list` 拉取，Pi 離線時顯示警告
7. **forward / security / command** 節點：只顯示 ICS 事件兩區塊，無 PWA 區塊。

### 表單與操作優化

8. **改派自動記錄**：`_setAssignedUnit` 變更後自動追加處置紀錄「指派變更：前進組 → 安全組」，新舊相同不寫。
9. **新增事件表單 z-index 修正**：`#event-overlay` / `#decision-overlay` 從 100 升至 220，不再被節點 modal（210）壓住。
10. **位置欄修正**：從節點 modal 開啟新增事件表單時，位置欄改填節點 MGRS 座標（若已放置地圖），不再誤填單位名稱。
11. **嚴重度 auto-sync**：危害類型下拉切換時，嚴重度 radio 自動同步 `NAPSG_EVENTS[type].severity`。
12. **送出後即時刷新**：`submitEvent()` 改為 `await poll()`，poll 完成後若節點 modal 仍開著，自動重繪事件頁，無需手動重整。

## v1.9 變更說明（對應 cmd-v0.10.0～v0.10.1）

### 地圖模組擴充（Phase 2.5）

1. **MGRS 搜尋欄**：設定面板新增獨立搜尋框，支援輸入 MGRS 或 WGS84（`lat, lng`）座標。三層自動補前綴：純數字串 → 補完整 GZD；2 字母+數字 → 補 Zone+Band；完整格式 → 直接解析。搜尋後 panTo 並展開事件 popup。
2. **永久節點放置**：設定面板新增「放置節點」按鈕群（收容組/醫療組/指揮部/前進組/安全組）。點擊後進入 `_nodePlaceMode`，在地圖點擊即固定 NAPSG 圓形節點（icon=`pin`）。位置存入 `map_config.outdoor.zones`，支援拖移。
3. **底圖固定 Grayscale**：移除 Leaflet layer control，底圖唯一為 protomaps Grayscale。
4. **節點視覺統一**：所有永久節點（收容組/醫療組等）統一深灰色（`#3a4149`），不再依 `_NODE_COLORS` 映射各隊顏色。僅事件 diamond 採 severity 燈號色。
5. **Tooltip 改為 hover-only**：`permanent: false`，游標移近才顯示標籤，避免地圖雜訊。

### 事件 severity 精細化

棄用舊版 `_CRITICAL_TYPES` 硬編碼集合（僅含 `drone`/`explosive`），改為每個 NAPSG 事件類型各自宣告 `severity` 欄位，全 22 種類型均已定義：

| Severity | 事件類型 |
|----------|---------|
| `critical`（紅）| 疑似爆裂物、無人機威脅、暴力事件、大量傷亡（MCI）、緊急病症 |
| `warning`（黃）| 不明人士、管制區異常、秩序問題、受困救援、QRF 出動、傳染疑慮、量能超載、隔離事件、通訊異常、撤離 |
| `info`（灰）| 人員狀況、設施異常、設備故障、資源調度、現場變化、危害回報、其他 |

`_placeOnLeaflet` 改用 `NAPSG_EVENTS[type]?.severity || 'warning'` 取得 severity，無需維護額外 Set。

## v1.8 變更說明（對應 cmd-v0.9.8～v0.9.9）

### 設計原則變更：回報人 ≠ 處理人

原 `EVENT_CATEGORIES` 以 ICS 單位為分類軸（收容組、醫療組…）。  
實際場景：**回報狀況的人，不一定是處理的人**（例如收容組回報「疑似爆裂物」，但處置責任在前進組）。  
因此改採 **NAPSG 危害類型** 為分類軸，單位資訊分為兩個獨立欄位：

| 欄位 | 意義 |
|------|------|
| `reported_by_unit` | 誰回報（metadata，不影響分類） |
| `assigned_unit` | 指派哪個單位處理（可為 null，由指揮官後續指定） |

### 主要變更項目

| # | 變更項目 | 說明 |
|---|----------|------|
| 1 | `NAPSG_EVENTS` 取代 `EVENT_CATEGORIES` | 22 種危害類型，依 NAPSG 標準分 6 群組：安全威脅、搜救行動、醫療緊急、收容照護、基礎設施、行動管理；每種類型有 `defaultAssigned` 建議指派單位 |
| 2 | 地圖事件放置改為長按 popup | 原本需先從 edit-bar 選類型再點地圖（兩步驟），現在改為：**長按地圖 650ms → popup 出現在長按位置 → Level 1 選群組 → Level 2 選類型 → 自動放置**；popup 包含回報單位下拉；不需離開地圖視野 |
| 3 | 雙擊地圖 = 藍色十字參考座標 | 雙擊地圖在點擊位置放置藍色十字 `✛`，僅作為座標參考用；中央上方顯示 MGRS 座標面板（可點擊 panTo）；與長按放置事件不衝突 |
| 4 | 座標面板持久顯示 | 有藍色十字時，座標面板持續顯示；放置事件後 3 秒提示（✓ 放置）結束後自動恢復顯示藍十字座標；拖移 marker 後 2.5 秒提示結束後同樣恢復；藍十字不因事件放置或 marker 拖移而消失 |
| 5 | `assigned_unit` DB 欄位 | `events` 表新增 `assigned_unit TEXT`；`PATCH /api/events/{id}` 白名單更新；事件處置 modal 可編輯指派單位 |
| 6 | DB 自動遷移 | `db.py` 新增 `_migrate_db()`：啟動時檢查 `assigned_unit` 是否存在，不存在則 `ALTER TABLE` 補欄；**不再需要刪 DB 升級 schema** |
| 7 | Leaflet popup 改用 DOM API | popup 按鈕改用 `L.DomUtil.create` + `L.DomEvent.on`（含 `stopPropagation`），取代 inline `onclick` 屬性；解決 `setContent` 更新後 click handler 失效問題 |
| 8 | Layer control 圖示 404 修正 | Leaflet 預設圖示（`images/layers-2x.png`）不在 repo 中；以 CSS 覆蓋為 `⊞` 文字圖示，消除 404 |

### 長按事件 popup 互動規格

```
長按地圖 650ms
  ↓
popup 出現（固定在長按座標）
  顯示：MGRS 座標 + 回報單位下拉 + 6 個群組按鈕
  ↓
點擊群組（例如「安全威脅」）
  ↓
同 popup 切換為該群組的事件類型清單 + 「← 返回」鈕
  ↓
點擊類型（例如「無人機威脅」）
  ↓
自動放置事件 marker、popup 關閉
上方顯示「✓ 放置 EV-MMDD-NNN MGRS: 51R TH xxxxx」3 秒
```

取消：按 popup 右上角 × 或點地圖其他位置關閉 popup，不放置任何事件。

### 操作模式對照

| 操作 | 行為 |
|------|------|
| 單擊地圖 | 無動作（或取消 marker 選取） |
| 雙擊地圖 | 在點擊位置放藍色十字參考座標 |
| 長按地圖（≥ 650ms） | 開啟事件回報 popup |
| 拖動開始 | 取消長按計時 |

> **觸控支援**：目前設計針對滑鼠操作。觸控長按為未來需求（已紀錄，尚未實作）。

## 第一部分　系統定位與設計哲學

### 1.1　核心概念：從資料到裁示的四層轉換

本系統的設計哲學是將分散的原始回報數據，逐層轉化為指揮官可直接裁示的決策資訊。所有介面設計、資料表設計、計算邏輯，都圍繞著這條轉換鏈展開：

| 層級 | 定義 | 由誰處理 | 對應介面 |
|------|------|----------|----------|
| 資料層 | 各組回報的原始事實：人數、床位、物資、傷患分類 | 各組 PWA 自動 / 計劃情報組代輸 / 感知系統 STT | 各組操作 PWA |
| 資訊層 | 交叉比對後的狀況意義：等待量上升、後送能力不足 | 計算引擎自動 / 計劃情報組補充 | 幕僚版儀表板（v6） |
| 態勢/預測層 | 5分鐘與15分鐘窗的風險判斷：8分後醫療進入警戒 | 計算引擎（SNAPSHOT + 趨勢引擎） | 幕僚版→指揮官版（v7/v8） |
| 裁示層 | 需指揮官決定的動作：核可補給、調派車輛 | 指揮官 | 指揮官版儀表板（v8） |

> **ℹ v1.1 整合說明**　v1.1 更新：§1.1 系統架構圖採用安全架構規格（v1.0）的更完整版本，補入 Pi Zero 2W Field Node。

```
【前進組 / 安全組】
Field Node (Pi Zero 2W)
PTT 錄音 → 本機 Whisper STT → 上傳指揮部
                    ↓
【收容組 / 醫療組】              【指揮部 Console】
手機 PWA                         Pi 500
   ↕ WebSocket / WSS              - ICS_DMAS 指揮部儀表板
各組 Pi ──────────────────→       - 民防感知系統 Console
(Pi 500)                          - SQLite（LUKS 加密，見第十七部分）
                                  - 幕僚版儀表板
                                  - 指揮官版儀表板（待建）
```

### 1.2　兩層儀表板架構

|  | **幕僚版（v6）** | **指揮官版（v8）** |
|--|-----------------|------------------|
| 主要使用者 | 計劃情報組、各組幕僚 | ICS 指揮官、副指揮官 |
| 資訊組織 | 按組別分欄（收容/醫療/安全/前進） | 按事件因果流（前進→CCP→醫療→後送；收容） |
| 核心任務 | 資料輸入、各組狀態監控、明細查詢、事件記錄 | 態勢判讀、資源裁示、決策追蹤 |
| 地圖位置 | 中央最大區（操作導向） | 第四列（驗證工具，不搶主角） |
| 事件呈現 | 完整事件列表，各組各自顯示 | 只顯示需要指揮官介入的裁示事項（合併為決策主題卡片） |
| 開發狀態 | 原型完成，待補輸入功能 | 原型完成（v8），待接真實資料 |

> *※ 幕僚版儀表板（v6）是「資料層→資訊層的中間介面」，不是指揮官工具。*
>
> *※ §1.2 各節點資料所有權：完整人員名單存各組 Pi SQLite（LUKS 加密），有網路時同步至指揮部 Pi 500 SQLite（LUKS 加密）。詳見第十七部分。*

## 第二部分　核心設計原則

| 原則 | 說明 |
|------|------|
| 先結論，後細節 | 主畫面先回答「現在是什麼局」「哪裡最危急」「接下來15分鐘會怎樣」「我現在要批什麼」 |
| 先態勢，再分工 | 指揮官先看跨組整合後的局勢，不自己在腦中重組分散卡片 |
| 按因果排，不按組織排 | 主畫面沿事件流與資源流展示，而非沿組織圖展示 |
| 時間新鮮度與可信度 | 每個模組均須同時顯示最後更新時間、新鮮度等級與預測信心度 |
| 現況＋趨勢＋預測 | 每一項關鍵數值同時有現況、趨勢方向與短窗預測，不停留在截面資料 |
| 門檻有預設值可調整 | 所有警示門檻有出廠預設值，可由管理介面調整，變更記錄於 AUDIT_LOG |
| 決策具備生命週期 | 一個決策主題可歷經 initial → revision/escalation → closure 多階段 |

> **ℹ v1.1 整合說明**　第四至九部分內容同 v1.0，此處列出關鍵摘要。完整規格以 v1.0 原文件為準。

### 時間新鮮度四級系統（第五部分）

| 等級 | 時間門檻 | 顯示樣式 | 處置意義 |
|------|----------|----------|----------|
| 🟢 即時 | < 3 分鐘 | 綠色圓點（實心） | 可視為當前可用資訊 |
| 🟡 注意 | 3–5 分鐘 | 黃色圓點（閃爍） | 需提高追蹤頻率 |
| 🔴 逾時 | > 5 分鐘 | 紅色圓點（閃爍） | 不可直接當成現況，需儘速驗證 |

### DECISION 決策主題生命週期（第九部分）

| decision_type | 觸發條件 | 對 severity 的影響 |
|--------------|----------|-------------------|
| initial | 第一次建立此決策主題 | 設定初始 severity |
| revision | 同等 severity 但調整建議動作 | severity 不變 |
| escalation | severity 提升（warning → critical） | severity 從 warning 升為 critical |
| closure | 決策主題結案：問題解決、資源到位、狀況解除 | severity 歸零，卡片移入已裁示記錄 |

> *※ severity 只能提升（revision 不降級），若情況好轉應建立 closure 而非 revision。*
>
> *※ closure 採「新建一筆 decision_type=closure 的記錄」，不修改現有記錄（Write-Freeze 原則）。*

> **ℹ v1.1 整合說明**　本部分整合自「ICS_DMAS 安全與網路架構規格 v1.0」§3、§4、§6、§7。

### 15.1　四種網路情境定義

| 情境 | 名稱 | 網路條件 | 通訊加密 |
|------|------|----------|----------|
| 情境 1A | 有公網，手機連同一 WiFi | 手機與各組 Pi 在同一 LAN | HTTPS + WSS（mkcert） |
| 情境 1B | 有公網，手機使用行動網路 | 手機透過 WireGuard VPN 連回指揮部 | WireGuard + HTTPS + WSS |
| 情境 2 | 無公網，自建 WiFi（Bridge Mode） | 所有 AP 設為 Bridge Mode，同一子網 | HTTPS + WSS（mkcert） |
| 情境 3 | 單機作業，無跨組網路 | 各組獨立運作 | 本機；QR code 為唯一態勢通道 |

> *※ 情境 2 所有 AP 必須設定為 Bridge Mode（同一子網），否則 WebSocket 無法跨 AP 連線。*
>
> *※ 情境 3 資料等網路恢復後透過三 Pass 對齊同步至指揮部（見第十四部分）。*

### 15.3　傳輸安全：mkcert 本地 CA

選用依據：作者 Filippo Valsorda（義大利裔美國人，前 Go 安全團隊），無中國關聯，開源，免費。

不使用 Let's Encrypt 原因：需要公開域名及網路驗證，情境 2、3 無法使用。

**憑證管理 SOP（演訓前）**

```bash
# 步驟 1：管理員 Mac 安裝 mkcert
brew install mkcert && mkcert -install

# 步驟 2：為各節點 IP 簽發憑證
mkcert 192.168.100.10  # 指揮部
mkcert 192.168.100.20  # 收容組 Pi
mkcert 192.168.100.30  # 醫療組 Pi

# 步驟 3：CA 根憑證安裝至所有手機/平板
# iOS：AirDrop rootCA.pem → 設定 → 一般 → VPN 與裝置管理 → 信任憑證
# Android：設定 → 安全性 → 安裝憑證

# 步驟 4：將憑證部署至各 Pi
scp 192.168.100.10+1.pem pi@192.168.100.10:~/certs/
scp 192.168.100.10+1-key.pem pi@192.168.100.10:~/certs/

# 步驟 5：啟動服務時載入憑證（見 §15.5）

# 步驟 6：全裝置連線測試
```

> *※ 憑證有效期預設 825 天，演訓前檢查是否過期。*

### 15.4　情境 1B：WireGuard VPN

選用依據：Jason Donenfeld（美國）開發，已併入 Linux kernel 5.6，無中國關聯。

| 設定項目 | 說明 |
|----------|------|
| 指揮部 Pi 500 開 WireGuard server | 監聽 UDP 51820 |
| 每台手機一組金鑰對 | 設定以 QR code 分發給使用者 |
| 若無固定公網 IP | 使用 DuckDNS（社群維護，免費）或 No-IP（美國） |
| 路由器設定 | 需開 UDP 51820 port forwarding |
| WireGuard 生效後 | 手機視為在同一虛擬 LAN，後續通訊與情境 1A/2 相同 |

### 15.7　PIN Lock（閒置自動鎖定）

| 項目 | 規格 |
|------|------|
| 閒置警告 | 閒置 **90 秒**後顯示全螢幕警告橫幅，提示即將鎖定 |
| 自動登出 | 警告後再 **30 秒**（即閒置起 **2 分鐘**）無操作，自動清除 session、跳回 PIN 輸入畫面 |
| 解鎖方式 | 輸入 4 位 PIN（不顯示明碼）；3 次錯誤觸發 30 秒冷卻 |
| 設定範圍 | 管理員可調整閒置門檻（最短 60 秒、最長 10 分鐘）；變更記錄於 AUDIT_LOG |
| 適用對象 | 所有登入角色；指揮官版儀表板與幕僚版均適用 |
| Phase | Phase 1 實作（與其他 UI 功能同期） |

> *※ PIN lock 計時器在任何滑鼠移動、鍵盤輸入、或觸控事件後重置。*

## 第十七部分　靜態加密與緊急銷毀（v1.1 新增）

> **ℹ v1.1 整合說明**　本部分整合自「ICS_DMAS 安全與網路架構規格 v1.0」§5。

### 17.1　方案選擇：LUKS

選用 LUKS（Linux Unified Key Setup），不使用 SQLCipher 或應用層加密。

| 原因 | 說明 |
|------|------|
| OS 層磁碟加密 | 覆蓋所有資料（SQLite、log、設定檔、暫存檔），不需修改應用程式碼 |
| Linux kernel 內建 | 無額外供應鏈風險 |
| 一致性 | 與民防輔助感知系統規格 §4.2 一致（cryptsetup luksErase） |

### 17.2　各節點 LUKS 設定

| 節點 | 加密對象 | 金鑰管理 |
|------|----------|----------|
| 指揮部 Pi 500 | OS 碟 + Data Cartridge（LUKS） | YubiKey（主）+ 備援密碼（雙人原則） |
| 各組 Pi | Data 分區（/data） | 演訓前設定 passphrase，任務結束後銷毀 |
| Field Node（Pi Zero 2W） | MicroSD | Panic 長按 5 秒快速格式化 |
| 測試機（Mac，Phase 1） | FileVault 全磁碟加密 | 系統偏好設定 → 隱私與安全性 |

### 17.3　Panic Wipe 機制

| 裝置 | 操作 | 效果 |
|------|------|------|
| Pi 500 | `cryptsetup luksErase` | 0.5 秒內 Header 抹除，資料永久遺失 |
| Field Node | 長按實體按鈕 5 秒 | MicroSD 快速格式化 |
| 安全設計要求 | Panic 按鈕需有防誤觸保護蓋 | |

> **⚠ 安全注意事項**　luksErase 操作不可逆，資料永久消失。演訓前必須確認所有資料已備份至安全媒介，或確認銷毀是蓄意且授權的決策。

### Pi Push 技術說明

> **v1.3 重寫**：取代原「Pi Read-Only API 技術說明」。

**Pi 端新增 table**：

- `current_state`：`(table_name, record_id, record_json, updated_at) PRIMARY KEY (table_name, record_id)`，在 `appendDelta()` 中 UPSERT，維持各 record 最新版本
- `push_queue`：`(id, pushed_at TEXT, records_json TEXT, sent INTEGER DEFAULT 0)`

**推送流程（每 60 秒）**：
1. 讀 `current_state` 全表 → JSON 化
2. 寫入 `push_queue`（sent=0）
3. 嘗試 `POST /api/pi-push/{unit_id}`（Bearer API key）
4. 成功 → `UPDATE push_queue SET sent=1`
5. 復線偵測：push 成功後，查 `sent=0` 舊 queue，按 pushed_at 順序補送
6. 定期清理：`DELETE FROM push_queue WHERE pushed_at < datetime('now', '-24 hours')`

**Command 端新增 table**：

- `pi_received_batches`：`(id, unit_id, pushed_at, received_at, records_json)`

**資料衍生**：Command 自行從收到的 records_json 衍生 aggregate counts（GROUP BY status），不依賴 Pi 端計算。

**syncTables 參考**（current_state 涵蓋的 table）：

| 組別 | table |
|------|-------|
| 收容組 | `persons`, `beds`, `resources`, `incidents`, `shifts` |
| 醫療組 | `patients`, `triages`, `incidents`, `shifts`, `transfers` |
